# Авторизация, multi-device sync и публичный лидерборд — Design Spec

**Дата:** 2026-06-03
**Статус:** Draft v1 — ждёт ревью пользователя
**Автор:** Boris (через брейнсторм с Сергеем)
**Связанный спек:** [2026-05-23-leaderboard-design.md](2026-05-23-leaderboard-design.md) (исходный серверный бэкенд + friends-лидерборд)

## 1. Зачем это вообще

У приложения три дыры, которые мешают ему стать чем-то большим, чем personal-tool:

1. **Нет синка между устройствами.** Идентичность сейчас = устройство: на машине генерится `api_secret`, лежит в Keychain, его хэш = аккаунт на сервере. Два ноутбука = два разных аккаунта, единой статы нет.
2. **Нет публичного лидерборда.** Есть только friends-лидерборд в приложении. Наружу показать нечего, виральности ноль.
3. **GitHub-интеграция — боль.** Руками выпусти PAT, вставь ключ, дай scope. Барьер на ровном месте.

Корень первых двух — отсутствие настоящей идентичности. Решаем через авторизацию. Третью чиним попутно: GitHub-вход выдаёт data-токен автоматически и убивает ручной PAT.

## 2. Главный принцип: трёхуровневый opt-in

Авторизация нужна **только** ради синка и шаринга. Кто этого не хочет — живёт локально и не отдаёт серверу ничего.

| Слой | Где живёт | Что включает | Аккаунт нужен? |
|---|---|---|---|
| Локальная стата (AI-траты) | Локально | Работает сразу после установки, ничего наружу | Нет |
| GitHub-стата (коммиты/LOC) | Локально, через GitHub OAuth | Убирает ручной PAT | Да (GitHub) |
| Синк между устройствами | Аккаунт (любой провайдер) | Единая стата на N машин | Да |
| Лидерборд друзей | Аккаунт, **в приложении** | Как сейчас | Да |
| Глобальный лидерборд | Аккаунт, **на сайте** | Отдельный публичный opt-in | Да |

Большинство юзеров остаётся на уровне 0 — про них на сервере нет ничего. Это и есть ответ на «легко, доступно любому, не собирать данные».

## 3. Архитектурный выбор

Рассмотрели три варианта идентичности:

- **A. Сервер-брокер OAuth, GitHub-first** — выбран.
- **B. Без провайдеров, привязка устройств по парному коду + recovery-фраза** — отклонён: не чинит PAT-боль и даёт легко-подделываемую идентичность (мусор в публичном лидерборде).
- **C. Децентрализованное хранилище в GitHub-репо юзера** — отклонён сразу: физически несовместимо с публичным лидербордом (нет центрального места для агрегации) и больше фрикции для юзера, не меньше.

Решение по A: данные в единой БД (она уже есть), сторонние провайдеры брокерит сервер, клиент остаётся провайдер-агностичным.

## 4. Модель идентичности

### 4.1 Серверная схема (эволюция текущей)

Сейчас `profiles.api_secret_hash` — это 1:1 «секрет = аккаунт». Расщепляем:

```sql
-- profiles: остаётся, секрет уезжает, добавляется флаг глобального opt-in
profiles (
  id, friend_code, display_name, avatar, avatar_mime,
  sharing_enabled  bool NOT NULL DEFAULT false,  -- шаринг с друзьями
  global_opt_in    bool NOT NULL DEFAULT false,  -- НОВОЕ: публичный лидерборд
  created_at, updated_at
)

-- НОВОЕ: привязанные внешние identity = «логин»
auth_identities (
  id            bigserial PRIMARY KEY,
  profile_id    bigint NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  provider      text NOT NULL,   -- 'github' | 'google' | 'yandex'
  provider_sub  text NOT NULL,   -- стабильный opaque id от провайдера
  handle        text,            -- github login и т.п., для отображения
  created_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (provider, provider_sub)
)

-- НОВОЕ: токены на устройство = multi-device
device_tokens (
  id            bigserial PRIMARY KEY,
  profile_id    bigint NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  token_hash    text NOT NULL,   -- sha256(device_token)
  device_label  text,            -- "MacBook Серёжи"
  last_seen_at  timestamptz,
  created_at    timestamptz NOT NULL DEFAULT now()
)
```

Один профиль → N device_tokens → multi-device решается тем, что логинишься на втором устройстве и получаешь второй токен на тот же `profile_id`. Старый `api_secret` становится одной из строк `device_tokens`.

### 4.2 Что сервер НЕ хранит

- **GitHub access-токен** (для тяги коммитов/LOC). Сервер получает его при OAuth, разово дёргает GitHub `/user` ради `sub`+login, пишет их в `auth_identities`, отдаёт токен клиенту и забывает. Токен живёт в Keychain на устройстве — как PAT сейчас, только выданный автоматически и отзываемый.
- **Email.** Не запрашиваем и не храним.

### 4.3 OAuth-брокер flow (token-mediating backend)

```
app                         сервер (aiuse)                провайдер
 │  ASWebAuthSession                                          │
 │ ── /api/auth/start?provider=github&challenge=H ──────────► │
 │                          redirect на authorize URL ──────► │
 │                                              юзер логинится │
 │                          ◄── /api/auth/cb/github?code ──── │
 │                          обмен code→gh_token,              │
 │                          GitHub /user → sub+login,         │
 │                          upsert profile+identity,          │
 │                          mint device_token, one-time code  │
 │ ◄── redirect burn://auth/callback?code=AUTH ────────────── │
 │ POST /api/auth/exchange {code, verifier} ────────────────► │
 │ ◄── {device_token, github_token?, friend_code, user_id} ── │
```

- `challenge/verifier` (PKCE-style) привязывают финальный обмен к тому экземпляру app, что начал флоу — защита от перехвата callback-URL.
- Client secret провайдеров живёт **только на сервере**, на устройстве никогда.
- Pending-сессии (challenge, provider, state, TTL) — в той же Postgres, без Redis/очередей.
- Добавить Google/Yandex = новый `case` в `/api/auth/cb/<provider>` на сервере. Релиз приложения не нужен.
- `burn://` — регистрируем custom URL scheme в `Info.plist`.

### 4.4 Два режима одного флоу

- **Не залогинен** → провайдер становится логином (создаёт/находит профиль).
- **Уже залогинен** (app предъявляет валидный device-token) → новая identity привязывается к существующему профилю.

### 4.5 Восстановление доступа

Исходный спек откладывал восстановление при потере Keychain. OAuth закрывает это бесплатно: потерял устройство — залогинился провайдером на новом, тот же профиль, данные на месте. Seed-фраза не нужна.

## 5. Multi-device sync

### 5.1 Проблема

Текущий снапшот-синк шлёт абсолютные значения за час, ключ `(user_id, hour_bucket)`, сервер делает upsert-replace. Для одного устройства ок. Для двух ломается:

> Ноут A за час потратил 1000 токенов, ноут B — 500. Истинный итог = 1500. Каждый шлёт своё абсолютное число, replace → побеждает тот, кто синканул последним. Получаешь 500 или 1000, но не 1500.

ccusage на каждой машине читает только локальные логи → данные устройств непересекающиеся. Значит правильно — атрибутировать по устройству и суммировать на сервере.

### 5.2 Решение: device_id в снапшотах

```sql
snapshots (
  profile_id    bigint NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  device_id     bigint NOT NULL REFERENCES device_tokens(id) ON DELETE CASCADE, -- НОВОЕ
  hour_bucket   timestamptz NOT NULL,
  tokens_input  bigint NOT NULL DEFAULT 0,
  tokens_output bigint NOT NULL DEFAULT 0,
  updated_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (profile_id, device_id, hour_bucket)
)
```

- Каждое устройство апсёртит только свои строки — идемпотентно, гонок нет.
- Все агрегаты (личные итоги, friends, global) — `SUM` по устройствам профиля.
- Клиент не шлёт `device_id` явно: сервер выводит его из Bearer-токена (device_token → device_id).
- Допущение: логи устройств не синхронизированы между собой (иначе двойной счёт). Это edge-case на стороне юзера, дефолт — непересекающиеся.

### 5.3 Device-linking без парных кодов

Добавить устройство = залогиниться тем же провайдером на втором устройстве. Сервер по `provider+sub` находит профиль и выдаёт новый device_token. Парные коды (минус варианта B) не нужны.

## 6. GitHub-слой через OAuth

### 6.1 Замена PAT

GitHub OAuth выдаёт access-токен автоматически (через тот же брокер, токен → Keychain клиента). Клиент продолжает тянуть GitHub напрямую — меняет PAT на OAuth-токен. Существующий код фетча (GraphQL + Contributor Stats) почти не трогаем.

### 6.2 Что шарится, а что нет

- Лидерборды (friends и global) ранжируют **по AI-токенам**, как сейчас. GitHub-коммиты/LOC на сервер **не уезжают** — это локальная фича отображения (popover, виджеты).
- Подключение GitHub нужно для двух вещей: идентичность/логин и автоматический data-токен вместо PAT. Ранжирование от GitHub не зависит.
- Следствие: даже подключив GitHub, ты не публикуешь коммиты — только опционально AI-токены через `sharing_enabled`/`global_opt_in`.

### 6.3 Scope

- **Дефолт: только публичные репы** (`read:user`, без `repo`).
- **Тогл «включая приватные репы»** → запрашиваем `repo`. Решает юзер осознанно.
- Тип токена — **классический OAuth App** (не GitHub App): не протухает, ведёт себя как PAT, минимум изменений клиента. Отозвали → 401 → клиент предлагает переподключить.

### 6.4 Миграция существующего PAT

На апгрейде PAT продолжает работать. Баннер предлагает подключить GitHub через вход. Подключился через OAuth → PAT заменяется на OAuth-токен, старый удаляем из Keychain. Не форсим.

## 7. Глобальный лидерборд на сайте

### 7.1 Где и как

- Глобальный — на сайте `aiuse.popovs.tech`, отдельной страницей. Друзья остаются в приложении.
- Публичное чтение: страницу видит любой без логина. Чтобы попасть в неё — явный opt-in.
- Метрика — AI-токены (input+output), периоды день/неделя/месяц (опционально all-time).

### 7.2 Два независимых флага видимости

```
sharing_enabled  → видят твои друзья (in-app friends-лидерборд)
global_opt_in    → виден в публичном лидерборде на сайте
```

Снапшоты пушатся, если включён **любой** флаг. Оба по умолчанию `false`. Включение `global_opt_in` — с явным предупреждением: handle, аватар и цифры станут видны публично.

### 7.3 Строка лидерборда

`rank · аватар · display_name/handle · tokens_total`. При GitHub-логине дефолт — github handle + аватар (реальные, труднее подделать). При Google/Yandex — заданный display_name.

### 7.4 Техника

- Эндпоинт `GET /api/global-leaderboard?period=` на существующем FastAPI: `SUM` снапшотов по всем `global_opt_in` профилям (по устройствам), группировка, ранжирование, top-N. С индексом на 500 юзеров — мгновенно.
- Страница — статическая в текущем landing-сетапе (nginx раздаёт статику), тянет JSON через fetch. Без Next.js, без отдельного фронт-сервиса.
- Кэш результата (пересчёт раз в N минут) добавляем, только если понадобится. Пока — живой запрос.

## 8. Геймификация

Минимум из данных, что уже есть, остальное — позже:

- **Ранг + дельта** ▲/▼/NEW — уже работает для друзей, переиспользуем на глобальном.
- **Стрики** (дни подряд с активностью) — считаются из снапшотов, дёшево.
- **Бейджи/ачивки/милстоуны** — откладываем до проверки, что соц-петля используется.
- **Пуши «тебя обогнали»** — остаются вырезанными.

## 9. Обработка ошибок

- OAuth отменён / провайдер лёг → app показывает ошибку, остаётся в текущем состоянии.
- GitHub-токен отозван (401) → предложить переподключить GitHub.
- Device-token отозван на сервере (выход на всех устройствах) → 401 → app падает в local-only + предложение перелогина.
- Оффлайн → существующая очередь `pending_snapshots`.
- Identity-конфликт (GitHub привязан к другому профилю) → явная ошибка, merge не делаем.

## 10. Тесты

- **Сервер (pytest + реальный Postgres-контейнер):** мокаем только внешний провайдер (token/user эндпоинты — это граница). Покрываем: выдачу device_token, линковку identity, конфликт identity, суммирование снапшотов по device_id, агрегацию глобального борда, гейтинг по `sharing_enabled`/`global_opt_in`.
- **Клиент (Swift):** построение и парсинг callback-URL, exchange-флоу, Keychain-стор, валидация (в стиле существующего `FriendCode`). Сам `ASWebAuthenticationSession` не юнит-тестим — изолируем логику вокруг.

## 11. Миграция данных

1. Существующий `api_secret` → строка в `device_tokens` (device_id = D0).
2. Старые снапшоты → backfill `device_id = D0` (миграция Alembic, ревью вручную перед коммитом).
3. При GitHub-логине на этой машине app предъявляет текущий токен → сервер привязывает GitHub-identity к существующему профилю, не создаёт новый.

## 12. Фазировка

Каждая фаза — отдельный план реализации, пишется перед началом фазы.

| Фаза | Содержание |
|------|------------|
| **1. Фундамент** | Сервер-брокер OAuth (GitHub), `auth_identities` + `device_tokens`, `device_id` в снапшотах + миграция, multi-device, привязка существующего профиля, GitHub-токен вместо PAT |
| **2. Глобал** | Публичная страница на сайте + `global_opt_in` + стрики и ранг-дельта |
| **3. Провайдеры** | Google + Yandex (только серверная правка) |
| **4. Бейджи** | Ачивки/милстоуны — если соц-петля взлетела |

## 13. Явно вырезано (YAGNI)

Слияние профилей (два аккаунта в один) · пуш-нотификации · GitHub-метрики в лидербордах · серьёзный анти-чит · глубокая модерация display_name (только базовый guard по длине + report-ссылка) · WebSockets/SSE (оставляем polling) · recovery-фраза (OAuth-релогин её заменяет) · мультипривязка провайдеров сверх необходимой линковки GitHub.

## 14. Открытые вопросы / гипотезы

- **Накрутка глобального лидерборда.** Есть вероятность, что реальные OAuth-аккаунты + существующие rate-limit'ы окажутся недостаточны, если кто-то начнёт скриптить usage. Полноценный анти-чит — за рамками MVP, добавим по факту проблемы.
- **all-time период** на глобальном борде — под вопросом, решаем в фазе 2 по факту.
- **Кэш глобального запроса** — добавляем, только если живой `SUM` начнёт тормозить (на 500 юзерах не должен).
